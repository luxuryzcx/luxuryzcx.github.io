## 前言

由于很多框架是 Java 写的，所以研究 Java 安全刻不容缓。这篇文章收录在「Java 安全 持续学习」系列中，用来沉淀对应主题的分析与记录。

## 正文

### 1. 定义

JEP 290 (Java Enhancement Proposal 290)** 是 Java 平台为了应对严重的反序列化漏洞而引入的一种   序列化数据过滤机制  。它通过在数据反序列化（即 `ObjectInputStream` 执行 `readObject()` 之前）的阶段增加一个校验层，决定是否允许特定类被反序列化。

### 2. 核心作用

- 实现“零信任”反序列化：改变了 Java 过去对序列化数据流“完全信任”的机制，强制对数据来源进行审查。
    
- 白名单防御：允许开发者通过配置白名单，只允许程序正常业务所需的类被实例化，从而直接屏蔽掉 `TemplatesImpl`、`ChainedTransformer` 等攻击载体。
    
-   防御深度提升 ：在类被加载到 JVM 内存之前即进行拦截，能够从根源上阻断 CC 链、7u21 等反序列化攻击链。
    

### 3. 工作原理

JEP 290 提供了一个    ObjectInputFilter   接口，其工作流程如下：

1. 序列化数据流传入  ：`ObjectInputStream` 接收到序列化数据。
    
2. 过滤器介入 ：在类对象被实例化前，`readObject()` 逻辑会询问过滤器：“这个类是否合法？”
    
3. 策略判断：
    
    - 允许 (ALLOWED)：类名存在于白名单中，继续反序列化。
        
    - 拒绝 (REJECTED)：类名存在于黑名单或未在白名单中，抛出 `InvalidClassException`。
        
4. 属性约束：除了类名过滤，还可以限制：
    
    - 数组维度：防止超大数组导致内存耗尽（DoS）。
        
    - 嵌套深度：防止递归结构导致的栈溢出。
        
    - 引用数量：防止恶意引用链造成的资源滥用。
        

### 4. 如何启用与配置

JEP 290 在 JDK 9 中正式引入，并被向后移植到了 Java 7 和 8 的后续更新版本中。

- 全局配置（通过系统属性）： 在 JVM 启动参数中直接设置：
    
    Bash
    
    ```c
    -Djdk.serialFilter=java.base/*;java.util/*;!com.evil.*;
    ```
    
    _(注：`!` 表示拒绝，`*` 表示通配符，此配置意为：允许基础类和集合类，拒绝所有 `com.evil` 包下的类)_
    
- 代码配置（针对特定流）**： 在代码中显式为某个 `ObjectInputStream` 设置过滤器：
    
    Java
    
    ```c
    ObjectInputStream ois = new ObjectInputStream(inputStream);
    ois.setObjectInputFilter(filter); // 设置自定义过滤器
    ```
    

### 5. 防御效果

- 通用性强：无需修改应用程序业务逻辑，即可通过配置大幅提升系统安全性。
    
- 补丁无法绕过的底线：即使应用依赖的第三方库（如 Commons-Collections）存在漏洞，只要 JEP 290 将危险类挡在门外，攻击者就无法构建完整的利用链。
    

总结：JEP 290 是现代 Java 系统应对反序列化攻击的**最后一道防线**。对于任何存在反序列化业务（如远程调用、缓存序列化、消息队列）的应用，开启 JEP 290 过滤都是安全配置中的强制标准。
